# Reproducibility Protocol

This document defines how to reproduce, verify, and extend the experiments in this repository without mixing published reference values with newly generated evidence.

## Result categories

Every reported number should be labelled as one of the following:

- **Published** — copied from the 2020 paper.
- **Reproduced** — generated by this repository with the original dataset and documented configuration.
- **Extended** — generated after changing the dataset, model, seed, preprocessing, architecture, or training procedure.

Do not present a published value as reproduced unless the corresponding run artifacts are available.

## Required inputs

Use the Corona NLP Text Classification files:

```text
data/raw/Corona_NLP_train.csv
data/raw/Corona_NLP_test.csv
```

Before training, record file hashes:

```bash
sha256sum data/raw/Corona_NLP_train.csv data/raw/Corona_NLP_test.csv
```

On Windows PowerShell:

```powershell
Get-FileHash data/raw/Corona_NLP_train.csv -Algorithm SHA256
Get-FileHash data/raw/Corona_NLP_test.csv -Algorithm SHA256
```

## Environment capture

Record at least:

- Operating system.
- Python version.
- PyTorch version.
- Transformers version.
- CUDA and cuDNN versions.
- GPU model and memory.
- Dataset file hashes.
- Git commit SHA.
- Configuration file used.

The training command automatically writes key software, label, dataset-size, and hardware metadata to `run_metadata.json`.

## Baseline execution

Run each paper-aligned configuration separately:

```bash
transformer-sentiment --config configs/bert.yaml --train-file data/raw/Corona_NLP_train.csv --test-file data/raw/Corona_NLP_test.csv --output-dir outputs/bert
transformer-sentiment --config configs/roberta.yaml --train-file data/raw/Corona_NLP_train.csv --test-file data/raw/Corona_NLP_test.csv --output-dir outputs/roberta
transformer-sentiment --config configs/albert.yaml --train-file data/raw/Corona_NLP_train.csv --test-file data/raw/Corona_NLP_test.csv --output-dir outputs/albert
```

## Minimum evidence for a reproduced result

A valid reproduced result should include:

1. `metrics.json`.
2. `classification_report.json`.
3. `predictions.csv`.
4. `run_metadata.json`.
5. The exact YAML configuration.
6. Dataset hashes.
7. Git commit SHA.
8. Hardware description.

## Metric policy

Use macro-F1 as the primary comparison metric. Also report:

- Accuracy.
- Macro precision and recall.
- Weighted precision, recall, and F1.
- Per-class precision, recall, and F1.
- One-vs-rest multiclass ROC-AUC when computable.

Do not select a model using test-set performance. Model selection must use the validation split only.

## Multiple-seed evaluation

A single seed is useful for paper alignment but not enough for a robustness claim. For stronger evidence, repeat each experiment using at least three seeds and report mean and standard deviation.

Suggested seeds:

```text
42, 123, 2020, 2026, 3407
```

Any multi-seed result is an extension unless the original paper explicitly used the same protocol.

## Sources of variation

Differences from the published scores may result from:

- Unspecified original dependency versions.
- Different pretrained checkpoint revisions.
- GPU nondeterminism.
- Dataset revisions or encoding differences.
- Validation split construction.
- Tokenizer implementation changes.
- Optimizer and scheduler implementation differences.
- Details absent from the paper.

A result that differs from the paper is not automatically incorrect. The important requirement is that the procedure and artifacts are transparent.

## Reporting checklist

Before adding a reproduced score to the repository, confirm:

- [ ] Dataset hashes are recorded.
- [ ] The exact Git commit is recorded.
- [ ] The configuration file is unchanged or the change is documented.
- [ ] The test set was not used for tuning.
- [ ] All output artifacts are retained.
- [ ] The result is labelled published, reproduced, or extended.
- [ ] Hardware and dependency versions are recorded.
- [ ] Failures and deviations are described.
